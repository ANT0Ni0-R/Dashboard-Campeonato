# New Mart Model

Create a new mart model for BU `$ARGUMENTS[0]` and entity `$ARGUMENTS[1]`.

## Conventions

1. **File path**: `models/bigquery/mart/[bu]/mrt_[bu]__[entity].sql`
2. **File name**: `mrt_[bu]__[entity].sql` (plural entity name)
3. **Schema**: Generated by macro as `mart_[bu]`

## Template

```sql
{{
    config(
        materialized = "table",
        meta = {'owner': '<user>'},
        tags = ["<lowest-frequency-source-tag>"]
    )
}}

-- Overview: <brief description of first source>
WITH source_a AS (
    SELECT *
    FROM {{ ref('stg_source__table_a') }}
)

-- Overview: <brief description of second source>
, source_b AS (
    SELECT *
    FROM {{ ref('stg_source__table_b') }}
)

-- Overview: <brief description of transformation>
, transformed AS (
    SELECT
        -- IDs (uid, surrogate keys, foreign keys)
        ---------------------------------
        -- Dates
        ---------------------------------
        -- Dimensions
        ---------------------------------
        -- Metrics
    FROM source_a
    LEFT JOIN source_b
        ON source_a.id = source_b.id
)

, final AS (
    SELECT *
    FROM transformed
)

SELECT *
FROM final
```

## Steps

1. Ask the user what sources/staging models this mart will consume
2. Read the referenced staging models to understand available columns
3. Create the mart model applying:
   - All refs at the top in simple CTEs with `SELECT *`
   - Descriptive CTE names
   - Leading commas
   - Explicit `AS` for all aliases
   - Column aliases prefixed with table alias in JOINs
   - `GROUP BY ALL` for aggregations
   - **BigQuery standard SQL only**: `SAFE_CAST`/`SAFE_DIVIDE`, `TIMESTAMP`/`DATE`/`DATETIME`, `DATE_DIFF`, `QUALIFY`, `ARRAY_AGG`, etc. Never hardcode table names — always `{{ ref() }}` (or `{{ source() }}` only when the upstream is an external mart not built by this project, e.g. `mart_portfel`).
   - **Column order**: always follow IDs → Dates → Dimensions → Metrics (separated by `---------------------------------` comments in transformation CTEs)
   - A `final` CTE before the closing `SELECT * FROM final`
4. Create/update the models YAML with column descriptions **and basic tests**. For any model under `models/mart/dashboard/`, the sibling `.yml` is mandatory and MUST declare `unique` + `not_null` on the model's key column. Apply the `document-sql-yml` skill conventions (column descriptions in pt-BR, `unique`/`not_null` on the key per its Passo 4.5, PII policy tags + `pii_level` for `materialized = table`).

## Rules
- No model header comment — straight from `config()` to `WITH`; docs live in YAML (the `-- Overview:` lines describe each CTE, not the model)
- Only use `{{ ref() }}` in mart (never `{{ source() }}`)
- Tag must correspond to the source with lowest update frequency
- Business rules and JOINs belong here
- Prefer `table` or `incremental` materialization
- No `ORDER BY`, no `DISTINCT`
- Use `dbt_utils.generate_surrogate_key()` for composite keys
- For reusable aggregate models (cubes/reports), expose only ADDITIVE measures (sums/counts) — compute ratios (ROAS/CPL/CVR/%, averages) at consumption, never pre-computed (a ratio doesn't re-aggregate). Pre-computed ratios OK only at single/final grain (1 row = 1 answer unit)
