# New Staging Model

Create a new staging model for source `$ARGUMENTS[0]` and table `$ARGUMENTS[1]`.

## Conventions

1. **File path**: `models/bigquery/staging/[source]/stg_[source]__[table_name].sql`
2. **File name**: `stg_[source]__[table_name].sql` (plural entity name)
3. **Schema**: Generated by macro as `staging_[source]`

## Template

```sql
{{
    config(
        materialized = "view",
        meta = {'owner': '<user>'},
        tags = ["<source-tag>"]
    )
}}

WITH bronze AS (
    SELECT
        -- IDs (uid, surrogate keys, foreign keys)
        ---------------------------------
        -- Dates (use SAFE_CAST, ensure America/Sao_Paulo timezone)
        ---------------------------------
        -- Dimensions (rename to snake_case)
        ---------------------------------
        -- Metrics
    FROM {{ source('<source_schema>', '<table_name>') }}
)

, silver AS (
    SELECT *
    FROM bronze
)

SELECT *
FROM silver
```

## Steps

1. Check if the source is already defined in a `_[source]__sources.yml` file in the staging directory
2. If not, create/update the sources YAML
3. Read the raw source table schema to understand available columns
4. Create the staging model applying:
   - `SAFE_CAST` for type conversions
   - `snake_case` column names
   - `{{ harmonize_emails() }}` for email fields
   - `UPPER()` for PMP fields
   - `{{ harmonize_phones() }}` for phone fields
   - `{{ harmonize_documents() }}` for CPF/CNPJ fields
   - Timezone conversion to `America/Sao_Paulo` for dates
5. Create/update the models YAML with column descriptions and policy tags for PII fields

## Rules
- No model header comment — straight from `config()` to `WITH`; docs live in YAML
- Only use `{{ source() }}` in staging (never `{{ ref() }}`)
- No JOINs in staging
- No aggregations in staging
- No business rules (those go in mart)
- Materialized as `view` by default
- **Column order**: always follow IDs → Dates → Dimensions → Metrics (separated by `---------------------------------` comments in bronze CTE)
